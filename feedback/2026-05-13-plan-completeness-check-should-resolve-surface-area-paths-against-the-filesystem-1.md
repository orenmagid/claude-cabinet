## Plan completeness check should resolve Surface Area paths against the filesystem

**Friction:** `/plan` defines the Surface Area format in `SKILL.md:152-167` and the completeness check in `:233-241` says "every file mentioned in Implementation appears in Surface Area. New files are marked `(new)`." But nothing enforces semantic correctness of the paths themselves. pib-db structurally validates the section format (header present, lines start with `- files:`/`- dirs:`) but never resolves the path against the working tree. Result: it's trivial — for a human OR an LLM — to file an action with fictional paths, which silently breaks `/execute-plans`'s conflict graph in two ways: false isolation (two plans declared non-conflicting because their fake paths don't overlap, but their real touched files do) OR missed conflicts (a plan's `(new)` extraction path overlaps with another plan's real file in the source it extracts from, but neither lists the source). In one session today I filed 12 actions and 14 of 25 paths were guessed wrong — pure invention from plausible-sounding component names. Found only because the user pushed back.

**Suggestion:** In `/plan`'s completeness check phase, after the user finalizes a draft, parse the Surface Area section, resolve each `- files:`/`- dirs:` entry against `process.cwd()` (or the worktree root), and surface any miss as a block-level issue unless `(new)` appears on the same line. Cheap, deterministic, project-agnostic. Would also catch typos and stale paths after renames.

**Session context:** Filing 12 e2e-finding actions from a verify:cheap run. Surface-area paths were guessed from plausible names rather than verified; QA-cabinet-style audit caught the rot.
